# Refactoring to pnpm Workspace and Env Sharing

## Summary

Refactored the `oa-1` repository into a single `pnpm` workspace, renamed core service directories, and implemented a symlink-based strategy for sharing development environment variables.

## The Problem

The project had multiple disconnected packages (`machinen-bridge`, `machinen-runner`, `dtu/github-actions`). We needed to:

1. Consolidate them into a single workspace for easier dependency management.
2. Rename services to shorter, clearer names (`bridge`, `runner`).
3. Share development `.env` and `.dev.vars` across services without duplication.

## Investigation & Timeline

- **Initial State:**
  - `machinen-bridge/`
  - `machinen-runner/`
  - `dtu/github-actions/`
  - Each had its own `.env` and separate `pnpm-lock.yaml`.
- **Attempts:**
  - Renamed `machinen-bridge` -> `bridge` and `machinen-runner` -> `runner`.
  - Created a root `pnpm-workspace.yaml` and `package.json`.
  - Consolidated environment variables into a root `.env` and `.dev.vars`.
  - Switched from a "load from root" script strategy to a "symlink" strategy based on user preference.
  - Renamed root `.env` to `.env.example` to ensure users explicitly set up their local environment.

## Discovery & Key Findings

- Using symlinks for `.env` files allows standard tools (like `dotenv` or `node --env-file`) to work without additional configuration, while maintaining a single source of truth at the repository root.
- Cloudflare Wrangler uses `.dev.vars` by default, so we added a shared version for that as well.

## Resolution

- **Workspace:** Root `pnpm-workspace.yaml` defines `bridge`, `runner`, and `dtu/github-actions`.
- **Renames:** Folders and `package.json` names updated to `bridge` and `runner`.
- **Env:**
  - Root has `.env.example` and `.dev.vars`.
  - Services have symlinks: `.env -> ../.env` and `.dev.vars -> ../.dev.vars`.
