# Mirroring GitHub API in DTU and Refactoring to Zod Package

## Summary

Refactored the Digital Twin Universe (DTU) to act as a proper source of truth by mirroring the GitHub Actions REST API. Transformed the `dtu/github-actions` directory into a structured TypeScript package with Zod-based configuration and official payload support.

## The Problem

The DTU was using a simplified, narrative-based communication flow that didn't accurately reflect GitHub's pull-based architecture. Specifically:

- GitHub API mocks were located in the Bridge instead of the DTU.
- Mock events used simplified JSON payloads instead of official GitHub schemas.
- Configuration was ad-hoc and lacked validation.

## Investigation & Timeline

- **Initial State:** Researching the `agent-ci-bridge` and `agent-ci-runner` showed the Bridge was acting as both an orchestrator and a partial GitHub API mock.
- **Architecture Shift:** Decided to move all GitHub API mocks to the DTU to ensure the Runner's container interacts with a "real" (simulated) environment.
- **Component Refactoring:**
  - Implemented `dtu/github-actions/src/server.ts` to mirror `GET /repos/{owner}/{repo}/actions/jobs/{job_id}`.
  - Updated `dtu/github-actions/src/simulate.ts` to "seed" job metadata into the DTU server memory before triggering webhooks.
- **Zod Integration:** Refactored the entire DTU folder into a package with a Zod-validated configuration layer.

## Discovery & Key Findings

- Mirroring the official GitHub API at the network level (pointing `GITHUB_API_URL` to the DTU) allows for zero-code changes inside the runner container, ensuring high fidelity.
- In-memory state in the DTU server is sufficient for the "simulate-once" flow used in the DTU.

## Resolution

The system now follows this flow:

1. `simulate.ts` seeds the DTU server and sends a webhook to the Bridge.
2. The Runner polls the Bridge for job metadata.
3. The Runner's container fetches the full "Plan" directly from the DTU server using official GitHub API paths.
